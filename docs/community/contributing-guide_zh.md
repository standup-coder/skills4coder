## 入门

### 先决条件
在为 Skills4Coder 做贡献之前，请确保您拥有：
- GitHub 账户
- Git 和 Markdown 的基础知识
- 您希望贡献的技能级别的理解（初学者、中级、高级、专家）

### 第一步
1. **派生存储库**
   点击 Skills4Coder 存储库页面右上角的 "Fork" 按钮

2. **克隆您的派生**
   ```bash
   git clone https://github.com/YOUR_USERNAME/skills4coder.git
   cd skills4coder
   ```

3. **创建分支**
   ```bash
   git checkout -b feature/your-contribution-type
   ```

4. **进行更改**
   遵循下面的贡献指南

5. **提交拉取请求**
   推送您的更改并向主存储库创建拉取请求

## 贡献方式

### 1. 添加新技能
向现有或新的技能清单添加相关技能：

**指南:**
- 确保技能符合目标级别的能力
- 包含清晰简洁的描述
- 尽可能添加相关的学习资源
- 遵循现有格式：`- [ ] 技能名称 - 描述`

**示例:**
```markdown
- [ ] Git 分支 - 了解如何在 Git 中创建、合并和管理分支
```

### 2. 改进现有技能
增强现有的技能描述或学习资源：

**指南:**
- 使描述更清晰、更准确
- 更新过时信息
- 添加或替换具有更好替代方案的学习资源
- 修复语法或拼写错误

### 3. 翻译
帮助将内容翻译成新语言：

**指南:**
- 为现有文档创建 `_zh.md` 或 `_es.md` 版本
- 在确保文化相关性的同时保持技术准确性
- 在翻译文档中使用一致的术语
- 在适当的地方包含语言导航

### 4. 文档
改进项目文档：

**关注领域:**
- 用户指南和教程
- 常见问题解答部分
- 贡献指南
- 社区资源
- 最佳实践文档

### 5. 代码和结构
改进项目结构和代码：

**示例:**
- 修复文档中的断开链接
- 改进文件夹组织
- 向文件添加缺失的元数据
- 为内容管理创建实用脚本

### 6. 错误报告
报告项目中的问题：

**错误报告中包含:**
- 问题的清晰描述
- 重现步骤
- 期望 vs. 实际行为
- 适用时的屏幕截图

### 7. 功能请求
提出新功能或改进建议：

**好的功能请求包括:**
- 问题陈述
- 建议的解决方案
- 对社区的好处
- 潜在的实施方法

## 行为准则

通过参与此项目，您同意遵守我们的[行为准则](../../CODE_OF_CONDUCT.md)。请在贡献之前阅读。

### 预期行为
- 尊重和包容
- 提供建设性反馈
- 对其他社区成员表现出同理心
- 尊重不同的观点和经验

### 不可接受的行为
- 任何形式的骚扰或歧视
- 煽动性或冒犯性评论
- 人身攻击
- 未经同意发布他人的私人信息

## 开发设置

### 必需工具
- Git
- 文本编辑器或 IDE
- 基本命令行工具
- 用于查看 Markdown 文件的网页浏览器

### 使用 Markdown
Skills4Coder 对所有文档和技能清单使用 Markdown：

**技能清单的标准格式:**
```markdown
# 类别标题

- [ ] 技能名称 - 技能的简短描述
- [ ] 另一个技能 - 另一个简短描述
- [x] 已完成技能 - 此项被勾选表示已完成

## 学习资源

1. [资源名称](https://example.com) - 资源描述
2. [另一个资源](https://example.com) - 另一个资源描述
```

## 提交更改

### 拉取请求流程
1. **更新您的派生**
   ```bash
   git checkout main
   git pull upstream main
   git checkout your-branch
   git rebase main
   ```

2. **验证您的更改**
   - 测试链接和引用
   - 检查格式和语法
   - 确保与现有内容的一致性

3. **提交您的更改**
   ```bash
   git add .
   git commit -m "更改的简短描述"
   git push origin your-branch
   ```

4. **创建拉取请求**
   - 转到 GitHub 上的 Skills4Coder 存储库
   - 点击 "Compare & pull request"
   - 填写 PR 模板
   - 提交以供审核

### 提交消息指南
- 使用现在时（"Add" 而不是 "Added"）
- 保持消息简洁但具有描述性
- 在有帮助时以前缀添加区域（例如，"Docs: Add user guide", "Skills: Update beginner list"）

**示例:**
```
✅ Add new security skills to advanced level
✅ 修复中级技能描述中的错别字
✅ 更新 API 开发的学习资源
```

### 拉取请求要求
- 更改必须与单个问题或改进相关
- 所有内容必须遵循项目的样式指南
- 链接应该是功能性的和适当的
- 更改应该为社区增加价值
- PR 描述应该解释所做的更改

## 文档标准

### 写作风格
- 使用清晰简洁的语言
- 尽可能使用主动语态
- 保持术语一致性
- 确保技术准确性

### Markdown 格式
- 使用适当的标题层次（# 到 ######）
- 用于列表的项目符号（- 用于无序列表）
- 用反引号格式化代码（`code`）
- 用适当的归属链接到外部资源

### 内容质量
- 确保事实准确性
- 保持内容与技能级别的相关性
- 提供实用的、可操作的信息
- 在必要时提供上下文

## 审核流程

### 初始审核
当您提交拉取请求时，将经历：
1. **自动检查**（如果已实施）
2. **项目维护者的手动审核**
3. **社区反馈**（鼓励）

### 审核标准
根据以下标准评估更改：
- **准确性**: 信息在事实上是正确的
- **相关性**: 内容适合目标受众和级别
- **清晰度**: 写作清晰易懂
- **价值**: 贡献增加了有意义的价值
- **一致性**: 遵循项目标准和格式

### 处理反馈
- 及时回复审核评论
- 及时进行请求的更改
- 如需澄清请提问
- 感谢审阅者的意见

### 批准流程
- 至少需要一位维护者的批准
- 在合并之前解决所有反馈
- PR 作者不应批准自己的 PR

## 社区准则

### 沟通
- 保持耐心和理解
- 尊重地提问
- 提供建设性反馈
- 庆祝他人的贡献

### 协作
- 在他人的想法基础上构建
- 大方分享知识
- 欢迎新人
- 培育包容的环境

### 持续改进
- 积极参与正在进行的讨论
- 深思熟虑地提出改进建议
- 在提交之前测试您的更改
- 从反馈和错误中学习

## 认可

### 贡献者致谢
在以下地方承认贡献者：
- 拉取请求合并消息
- 项目发布说明
- 文档中的特殊贡献者部分

### 贡献级别
- **社区贡献者**: 个人贡献
- **核心贡献者**: 多个重要贡献
- **维护者**: 持续的项目监督和管理

## 获得帮助

### 遇到困难时
如果您需要帮助进行您的贡献：
1. **检查现有文档**
2. **搜索过去的问题以查找类似问题**
3. **如果您找不到答案，请在新问题中提问**

### 导师制
- 鼓励有经验的贡献者指导新人
- 加入讨论并提供指导
- 分享学习资源
