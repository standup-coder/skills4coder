## 入门

### 先决条件
在为 Vibe Coding 做贡献之前，请确保您拥有：
- GitHub 账户
- Git 和 Markdown 的基础知识
- 对 Vibe Coding 哲学和原则的理解
- 正念、创意或协作编码实践的经验

### 第一步
1. **派生存储库**
   点击 Vibe Coding 存储库页面右上角的 "Fork" 按钮

2. **克隆您的派生**
   ```bash
   git clone https://github.com/YOUR_USERNAME/vibe-coding-skills.git
   cd vibe-coding-skills
   ```

3. **创建分支**
   ```bash
   git checkout -b vibe-enhancement/your-contribution-type
   ```

4. **进行更改**
   遵循下面的贡献指南

5. **提交拉取请求**
   推送您的更改并向主存储库创建拉取请求

## 贡献方式

### 1. 添加新 Vibe 技能
添加体现 Vibe Coding 哲学的相关技能：

**指南:**
- 确保技能与 Vibe Coding 原则一致
- 包含清晰、鼓舞人心的描述
- 尽可能添加相关活动
- 遵循现有格式：`- [ ] 技能名称 - 描述`

**示例:**
```markdown
- [ ] 正念代码审查实践 - 以同理心和学习重点进行代码审查
```

### 2. 改进现有技能
增强现有的技能描述或活动：

**指南:**
- 使描述更具启发性和可操作性
- 更新过时信息
- 添加或替换具有更好替代方案的活动
- 修复语法或拼写错误

### 3. 翻译
帮助将内容翻译成新语言：

**指南:**
- 为现有文档创建 `_zh.md` 或 `_es.md` 版本
- 在确保文化相关性的同时保持哲学准确性
- 在翻译文档中使用一致的术语
- 保持 Vibe Coding 的鼓舞人心的语调

### 4. 文档
改进项目文档：

**关注领域:**
- 用户指南和教程
- 哲学解释
- 活动描述
- 最佳实践文档
- 社区资源

### 5. 活动与练习
创建培养 Vibe Coding 技能的新活动：

**指南:**
- 设计真正增强快乐和正念的活动
- 确保活动具有包容性和可访问性
- 包含清晰的说明和预期结果
- 设计活动时考虑不同技能水平

### 6. 代码和结构
改进项目结构和代码：

**示例:**
- 修复文档中的断开链接
- 改进文件夹组织
- 向文件添加缺失的元数据
- 为内容管理创建实用脚本

### 7. 错误报告
报告项目中的问题：

**错误报告中包含:**
- 问题的清晰描述
- 重现步骤
- 期望 vs. 实际行为
- 适用时的屏幕截图

### 8. 功能请求
提出新功能或改进建议：

**好的功能请求包括:**
- 问题陈述
- 建议的解决方案
- 对社区的好处
- 潜在的实施方法

## 行为准则

通过参与此项目，您同意遵守我们的[行为准则](../../CODE_OF_CONDUCT.md)。请在贡献之前阅读。该项目旨在培育一个欢迎和积极的社区，与 Vibe Coding 原则保持一致。

### 预期行为
- 保持尊重和包容
- 以同理心提供建设性反馈
- 对其他社区成员表示同情
- 尊重不同的观点和经验
- 庆祝差异和独特观点

### 不可接受的行为
- 任何形式的骚扰或歧视
- 煽动性或冒犯性评论
- 人身攻击
- 未经同意发布他人的私人信息
- 对他人经历或感受的蔑视对待

## 提交更改

### 拉取请求流程
1. **更新您的派生**
   ```bash
   git checkout main
   git pull upstream main
   git checkout your-branch
   git rebase main
   ```

2. **验证您的更改**
   - 测试链接和引用
   - 检查格式和语法
   - 确保与 Vibe Coding 哲学的一致性
   - 确认内容鼓舞人心且可操作

3. **提交您的更改**
   ```bash
   git add .
   git commit -m "Vibe 为中心的更改的简短描述"
   git push origin your-branch
   ```

4. **创建拉取请求**
   - 转到 GitHub 上的 Vibe Coding 存储库
   - 点击 "Compare & pull request"
   - 填写 PR 模板
   - 提交以供审核

### 提交消息指南
- 使用现在时（"Add" 而不是 "Added"）
- 保持消息简洁但具有描述性
- 在有帮助时以前缀添加区域（例如，"Docs: Add user guide", "Skills: Update beginner list"）
- 强调更改的积极、Vibe 增强方面

**示例:**
```
✅ 添加正念呼吸练习到初学者技能
✅ 在中级水平增强快乐协作技巧
✅ 改进创意编码探索的活动
```

### 拉取请求要求
- 更改必须与单个问题或改进相关
- 所有内容必须与 Vibe Coding 原则一致
- 链接应该是功能性的和适当的
- 更改应该为社区增加价值
- PR 描述应该解释更改的积极影响

## 文档标准

### 写作风格
- 使用积极、鼓舞人心的语言
- 尽可能使用主动语态
- 保持术语一致性
- 确保与 Vibe Coding 的哲学一致性
- 使用包容性语言，欢迎所有经验水平

### Markdown 格式
- 使用适当的标题层次（# 到 ######）
- 用于列表的项目符号（- 用于无序列表）
- 用反引号格式化代码（`code`）
- 用适当的归属链接到外部资源

### 内容质量
- 确保准确性同时保持启发性
- 保持与 Vibe Coding 哲学相关的内容
- 包含实用、可操作的信息
- 在必要时提供上下文
- 在整个过程中保持振奋人心的语调

## 审核流程

### 初始审核
当您提交拉取请求时，将经历：
1. **自动检查**（如果已实施）
2. **项目维护者的手动审核**
3. **社区反馈**（鼓励）

### 审核标准
根据以下标准评估更改：
- **一致性**: 内容符合 Vibe Coding 哲学
- **启发性**: 写作激励和鼓舞读者
- **可操作性**: 信息实用且可实施
- **价值**: 贡献为 Vibe 开发增加有意义的价值
- **一致性**: 遵循项目标准和语调

### 处理反馈
- 及时回复审核评论
- 及时进行请求的更改
- 如需澄清请提问
- 感谢审阅者的意见

### 批准流程
- 至少需要一位维护者的批准
- 在合并之前解决所有反馈
- PR 作者不应批准自己的 PR

## 社区准则

### 沟通
- 保持耐心和理解
- 尊重地提问
- 以同理心提供建设性反馈
- 庆祝他人的贡献

### 协作
- 在他人的想法基础上构建
- 大方分享知识
- 以热情欢迎新人

### 持续改进

在以下地方承认贡献者：
- 拉取请求合并消息
- 项目发布说明

### 贡献级别

- **Vibe 建设者**: 多个有意义的贡献
- **Vibe 领导者**: 对项目方向的重大影响

3. **如果您找不到答案，请在新问题中提问**

- 加入讨论并提供指导
- 分享 Vibe Coding 学习资源
